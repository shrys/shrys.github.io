MARSMonitoringWeb

  • It is a 3-tier business architecture web application to monitor files
  • Front end designes in HTML, CSS, BootStrap with JavaScript and jQuery controls
  • C#, JavaScript have extensively been used in this module's functioning
  • ASP.NET Web Pages, MVC frameworks have been used to bind data from back-end (Third party APIs DHTMLX, Infragistics, Telerik used)
  • PL/SQL in Oracle database has been used to fetch, update, edit and delete records of tables
  • BusinessObject class is used to get/set data from/to database
  • UNC paths are accessed programatically to view files in it
  • Security is ensured by making all Page classes child of BasePage
  • LINQ statements make datasource processing much simpler

GPGFileWatcher

  • It is a Windows Service written completley in C#
  • It takes entries from an XML file to monitor certain folder
  • Each folder is associated with its own password and in the folder .gpg files can be created
  • All .gpg files are secure and can only be opened by the intended person
  • GpgAPI library has been used in this module to decrypt the incoming files
  • Logs are maintained by the Windows Service to check its timely operations
  • After the file is detected and decrypted by the service, it is moved to an 'Archive' folder

PDFFileWatcher

  • It is a Windows Service written completley in C#
  • It monitors the whole storage and is deployed on production server directly
  • The service monitors file creation and deletion operations in the server and keeps track of which logged on user did it
  • The requirement stated that .pdf files are to be monitored
  • Upon any error and every 24 hours, when the service is running, logs will be mailed to an email address mentioned in App.config
  • Logs are maintained by the Windows Service to check its timely operations
  • A .NET library called Cassia is used in this module to know which logged on user is doing what operation to which .pdf file

PDFFileWatcher

  • It is Windows console application written in C#
  • The console application is used by the team to backup files from production server to backup server
  • It has two modes: standalone(takes source and destination from an XML file) and arguments(takes source and destination as arguments)
  • The requirement stated that .pdf files are to be monitored
  • Upon any error and after running application logs will be mailed to an email address mentioned in App.config
  • Logs are maintained by the Windows Service to check its timely operations
  • A .NET library called Zeta Long Paths is being used to overcome the Windows limitation where not more than 260 characters are allowed in a file path
  • The console application backs up files incrementally
